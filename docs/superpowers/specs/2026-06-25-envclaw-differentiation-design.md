# 环保管家客户端差异化设计

- **日期**：2026-06-25
- **状态**：设计迭代中，原型已确认
- **原型**：见同目录 `prototype-*.html`（工作台 / 向导 / 平台管理 / 智能问数）

---

## 1. 背景与问题

当前客户端基于 hermes-studio 二次开发（Electron，多端通用）。已做的优化包括：简化 Hermes 安装流程、更换核心组件下载源、内置 agent-browser CLI、内置 IPP-AIR 操作指南 skill。

**核心问题**：领导评价"差异化太少，本质是给 Hermes 套了个 UI"。具体痛点：

1. 入口仍是空白对话框，与官方 Hermes 无本质区别，预置功能只是多了几个可见入口。
2. 用户操作繁琐：创建无人值守任务仍要手写大段提示词，门槛高于直接用 Hermes。
3. 各平台（mapairs、国控站、OA、通讯工具）登录靠手动，每次重复操作。
4. 缺乏环保行业的垂直化体验，感觉是"通用 AI"而非"环保专业工具"。

**目标用户**：内部环保业务人员优先，逐步外推到外部客户（企业/政府）。这类用户熟悉环保业务但不熟悉 AI 工具。

**高频场景**：当前实际业务有两类——**数据采集**与**数据播报**。第一版优先落地数据播报。

**一句话定位**：
> 打开就能用的环保智能管家——把环保行业 know-how 沉淀进平台配置，靠"选平台 → 选功能 → 自动组装提示词"完成自动化任务，而非手写提示词。

注意：目标不是"零提示词"（不现实），而是**把提示词内化为平台配置 + 分步引导**——平台一次配好操作知识，建任务时点选即自动组装。

## 2. 总体方案：工作台为主体 + 智能问数平级入口

顶栏提供两个**平级**入口：

```
┌─────────────────────────────────────────────────────┐
│ 🌿 数智大气·环保管家   [工作台] [智能问数]   🔔 ⚙️ 👤 │
├─────────────────────────────────────────────────────┤
│  工作台模式：场景卡片 + 任务列表 + 平台管理 + 概览     │
│  智能问数模式：接入数智大气现有的流式对话服务（独立页面）│
└─────────────────────────────────────────────────────┘
```

**关键原则**：

- **工作台**是本产品的主体，承载场景卡片、值守任务、平台管理、概览。
- **智能问数**是顶栏平级入口，直接接入数智大气**现有的流式对话服务**（很可能以内嵌网页方式集成）。它是一个**完全独立的页面**，本项目不实现其对话逻辑，只负责入口与集成。
- 工作台创建的值守任务，可从**任务详情进入一个针对该任务的对话页面**（走 Hermes 对话能力），相当于围绕这个任务追问、调整、排障。这才是"对话"能力在本产品里的落点——绑定到具体任务，而非一个空白全局对话框。
- 切换模式不丢失状态（Pinia 持久化）。

**为什么是 C 而不是 A/B**：

- 方案 A（仅在现有对话框架上加向导弹窗）差异化不足，仍是 Hermes 皮肤。
- 方案 B（纯工作台、完全去掉对话能力）差异化强但牺牲了对话。Hermes 的对话能力仍有价值，但落点不应是一个空白全局对话框。
- 方案 C 把对话能力**绑定到具体值守任务**（任务详情页内对话），既保留了 Hermes 对话价值，又让它服务于明确的业务上下文；同时顶栏「智能问数」接入数智大气自有的对话服务作为另一条平级入口。差异化与能力完整性兼顾，且可渐进式迭代。

## 3. 模块设计

系统拆分为相互独立、通过明确接口协作的单元。每个单元可独立理解与测试。

### 3.1 工作台首页（Workstation Home）

**职责**：业务人员的默认落地页，把"能做什么"和"正在做什么"一屏呈现。

**组成**（自上而下）：

1. **欢迎概览** — 个性化问候 + 日期/单位 + 统计卡。
   - 本期仅"运行中任务""今日成功率"两张卡用真实数据（来自 jobs store）。
   - "待处理预警""今日数据采集"两张卡**置灰标注「待接入」**，数据如何采集/入库/持久化尚未确定，先占位不显示假数据。
2. **业务场景卡片** — 4 张卡片，本期只开放**数据播报**（可点击，进入创建向导）。
   - 数据采集、异常预警、报告生成三张做成**明确不可点击的"即将推出"态**（置灰 + `cursor: not-allowed`），让用户一眼看出功能未完成，而非误以为可用。
   - 不再有"智能问数"场景卡（智能问数已上移为顶栏平级入口）。
3. **活跃任务列表** — 运行中/暂停任务一览，含频率、推送渠道、上次运行时间、状态 pill、快捷操作（立即运行 / 更多）。点击任务进入**任务详情**（含针对该任务的对话页）。
4. **平台条** — 各平台连接状态一目了然，点击进入平台管理。

**左侧导航**：工作台首页、值守任务、平台管理、技能库、运行历史。**不含"智能问数"**（它在顶栏）。

**业务场景范围说明**：当前实际业务只有两类——数据采集与数据播报。第一版仅落地**数据播报**，其余为占位。

**依赖**：jobs store（任务数据）、platforms store（连接状态）、概览统计接口。

**新增 vs 复用**：新增 `WorkstationHome.vue` 及子组件；任务数据复用现有 `stores/hermes/jobs.ts`。

### 3.2 创建向导（Task Wizard）

**职责**：消除手写提示词的摩擦。核心思路是把环保 know-how 沉淀到**平台配置**里（见 3.3），新建任务退化为"选平台 → 选功能 → 自动组装提示词与技能"。

**模式 A — 模板向导（单页表单，本期落地"数据播报"）**：

字段顺序（自上而下，单页，不分步）：

1. **任务名称** — 文本输入。
2. **数据平台** — 选择一个已在「平台管理」配置好的平台。选中后，该平台预定义的**功能**自动列出。
3. **功能**（可多选）— 勾选功能（如"小时播报"）。每勾选一个：
   - 把该功能的**专属提示词**追加进"已组装提示词"预览。
   - 把该平台配置的**技能**自动加到本任务。
4. **执行提示词** — 分两部分：
   - **已组装提示词（只读预览）** = 模板角色基底 + 平台操作提示词 + 已选功能提示词，随勾选实时增减，结构清晰不会乱。
   - **补充说明（可编辑）** = 用户在此追加自定义要求，与组装部分分离、可追溯。
5. **执行频率** — 预设选项 + cron 表达式。**直接复用现有 `SchedulePicker` 组件**。
6. **推送平台** — 复用现有 `CreateGuardTaskModal` 的 `platformList`。主要用于**标记任务归属**，不保证实际投递生效。
7. **可选技能**（单/多选，放最后）— 数据来自技能管理；平台已带的技能在此预填，用户可增删。

**提示词组装公式**：
```
最终 prompt = 模板角色基底（模板自带，如"你是数据播报助手…"）
            + 平台操作提示词（来自所选平台）
            + Σ 已选功能提示词（来自所选功能）
            + 用户补充说明（手填，可选）
```

**模式 B — AI 引导**（自主规划，适合需求不在模板内或用户不确定如何配置时）：

- 类似 Claude Code 的交互：AI 一次只问一个问题，提供可点选的预设选项，也接受自由输入。
- AI 过程中注入环保领域知识（识别"平顶山"为城市、"PM2.5"为指标等）。
- 引导结束后给出**完整方案摘要卡**，用户可"调整"或"确认创建"。

**两种模式的边界**：模板向导覆盖固定结构化场景（本期=数据播报）；AI 引导作为模板无法覆盖时的兜底与自由路径。向导面板顶部一键切换。

**形态**：向导为**右侧滑出面板**（非弹窗），左侧任务列表保持可见，便于对照。

**依赖**：场景模板定义（见 3.6）、platforms store（列出平台、功能、平台技能）、技能管理（技能列表）、jobs store（创建任务）、AI 引导后端接口。最终仍产出现有 `CreateJobRequest`（`name/schedule/prompt/deliver/skills/repeat`）。

**新增 vs 复用**：新增 `TaskWizard.vue`（含 TemplateWizard / AiGuide 两个子组件）；最终落库复用现有 job 创建 API；频率选择复用 `SchedulePicker`，推送平台复用现有 `platformList`。现有 `CreateGuardTaskModal.vue` 的逻辑可迁移为模板向导的一部分。

### 3.3 平台管理（Platform Management）

**职责**：集中管理每个数据平台的**凭据 + 操作知识 + 技能 + 功能**，把环保 know-how 沉淀到平台配置，供新建任务直接消费。配置一次，人人会用。

**一个平台包含四部分**：

1. **账号凭据**（多账号）— 账号/密码等，做到"配置一次，自动登录"。
2. **平台操作提示词**（×1）— 这个平台怎么登录、页面结构、通用操作约定。
3. **平台技能**（单/多选）— 数据来自技能管理；选了任务就自动带上。
4. **功能**（多个）— 平台特有的操作（如"小时播报"），每个功能 = 名称 + **专属提示词**（怎么找到并操作这个功能）。

**数据模型**：

```ts
interface Platform {
  id: string
  type: 'mapairs' | 'national_station' | 'oa' | 'feishu' | 'dingtalk' | 'custom'
  name: string                 // 用户自定义，如"数智大气"
  url?: string
  operationPrompt: string      // 平台操作提示词（怎么登录/找页面）
  skills: string[]             // 平台技能 id，来自技能管理，单/多选
  functions: PlatformFunction[]// 平台功能列表
  accounts: PlatformAccount[]  // 多账号
}

interface PlatformFunction {
  id: string
  name: string                 // 如"小时播报"
  prompt: string               // 该功能专属提示词
}

interface PlatformAccount {
  id: string
  name: string                 // 如"平顶山账号"
  credentials: {
    type: 'password' | 'api_key' | 'webhook' | 'cookie'
    username?: string          // 加密存储
    password?: string          // 加密存储
    apiKey?: string            // 加密存储
    webhookUrl?: string
    extra?: Record<string, string>
  }
  status: 'connected' | 'expired' | 'error' | 'pending'
  lastLogin?: string
  lastError?: string
  autoRefresh: boolean         // 会话过期是否自动用凭据重登
}
```

**与新建任务的联动**（核心差异化）：新建任务选中某平台后——
- 该平台的**功能列表**自动跳出；
- 每勾选一个功能，把功能 `prompt` 追加进任务的"已组装提示词"，并把平台 `skills` 自动加到任务；
- 平台 `operationPrompt` 作为组装提示词的固定一段（见 3.2 组装公式）。

**安全设计**：

- 凭据以 AES-256 加密存储于本地 SQLite。
- 前端仅显示掩码（`••••••••`），不回传明文。
- 会话过期且 `autoRefresh` 为真时，用仍有效的凭据自动重新登录。

**多账号**：同一平台可配置多个账号（如多个地区的 mapairs 账号），新建任务时按平台列出可选账号。

**账号 vs 平台级配置的职责划分**（避免入口重叠）：

- **账号**（多个、平级）→ 在**平台卡片的账号子行**管理：添加账号、每行编辑/删除。点账号"编辑"弹小抽屉，只编这一个账号的凭据。
- **平台级配置**（操作提示词、平台技能、功能，平台唯一一份）→ 在平台**"配置"抽屉**管理。**抽屉不含账号**。

**两层入口**：

1. 工作台底部平台条（展示连接状态 + 快捷入口，名称统一为"平台管理"）。
2. 独立"平台管理"页（平台卡片列出账号子行 + 平台级配置摘要；"配置"抽屉管平台级三部分；账号编辑用独立小抽屉）。

**依赖**：本地加密存储（SQLite + AES-256）、技能管理（技能列表）、各平台登录的后端能力（经 agent-browser / Hermes 工具执行实际登录）。

**新增 vs 复用**：新增 `PlatformManagement.vue`、`platforms` store、加密存储后端模块、`platforms` API。与现有 `auth.ts`（应用自身用户体系）相互独立——平台凭据是"被自动化操作的外部站点凭据"，不要与应用登录混淆。

### 3.4 智能问数（Smart Query）

**职责**：自然语言查询环保数据。作为**顶栏平级入口**，接入数智大气**现有的流式对话服务**。

**接入方式**：智能问数是一个独立的流式对话服务，很可能以**内嵌网页**方式集成进客户端。本项目**不实现其对话/分析逻辑**，只负责：顶栏入口、页面容器、与该服务的集成（如鉴权透传、URL 装载）。具体集成形态待与该服务团队确认。

**与任务详情对话的区别**：

- **智能问数**（顶栏）= 数智大气自有的通用数据问答服务，全局、独立页面。
- **任务详情对话**（见 3.5）= 围绕某个具体值守任务的对话，走 Hermes 能力，绑定任务上下文。

两者是不同的对话入口，不要混淆。

**新增 vs 复用**：新增顶栏入口与页面容器；对话逻辑由外部服务提供，不在本项目实现。

### 3.5 任务详情与任务对话（Task Detail & Chat）

**职责**：每个值守任务点开后进入详情页，详情页内提供一个**针对该任务的对话页面**。

**交互**：

- 任务列表/活跃任务点击任意一项 → 任务详情（配置、运行历史、推送记录）。
- 详情页内可进入**任务对话**：围绕这个任务追问、调整配置、排查为什么某次没成功等。这是 Hermes 对话能力在本产品里的落点——绑定具体任务上下文，而非空白全局对话框。
- 可参考现有 Hermes 对话页面实现。

**依赖**：jobs store（任务数据）、Hermes 对话运行时（`run-chat` / agent-bridge）。

**新增 vs 复用**：新增任务详情页与任务对话入口；对话能力**复用现有 Hermes 对话运行时**，不新造对话内核。

### 3.6 场景模板系统（Scenario Templates）

**职责**：把硬编码的值守机器人改为可配置的模板库，驱动场景卡片与模板向导。

**现状**：当前 6 个值守机器人硬编码在 `GuardPanel.vue` 中，文案未走 i18n。

**改进**：

- 模板定义为结构化数据（场景类型、图标、描述、向导步骤、各步骤预设选项、prompt 模板）。
- **第一版只需落地"数据播报"模板**；数据采集、异常预警、报告生成先占位（场景卡置为不可点击），模板后续补齐。
- 模板向导（3.2 模式 A）的步骤与预设值由模板定义驱动。
- 模板自带**角色基底提示词**（如"你是数据播报助手…"），作为组装提示词的第一段。

**依赖**：被工作台首页（场景卡片）与创建向导（模板）消费。

**新增 vs 复用**：新增模板定义模块；重构 `GuardPanel.vue` 的硬编码数据为模板驱动，并补齐 i18n。

## 4. 与官方 Hermes 的差异化总结

| 维度 | 官方 Hermes | 本产品 |
|------|------------|--------|
| 首页 | 空白对话框 | 业务场景看板 + 数据概览 |
| 创建任务 | 手写提示词 | 选平台+功能自动组装提示词与技能 |
| 平台登录 | 每次手动 | 配置一次，自动登录 + 多账号 |
| 平台知识 | 无 | 平台沉淀操作提示词+技能+功能，任务直接复用 |
| 行业知识 | 通用 AI | 内置环保领域预设（地市、国标指标、平台） |
| 智能问数 | 无 | 顶栏平级入口，接入数智大气自有对话服务 |
| 任务对话 | 空白全局对话框 | 对话绑定到具体值守任务（任务详情内） |
| 值守模板 | 无 | 可配置场景模板库 |
| 用户取向 | 开发者向 | 业务人员向，点选为主 |

## 5. 设计语言

对齐项目现有 Naive UI 主题（见 `packages/client/src/styles/theme.ts`）：

- **配色**：黑白灰单色系（暗色 `--bg #1a1a1a`、卡片 `#2a2a2a`、主色 `#e0e0e0`）。状态色克制使用——绿 `#66bb6a`（正常/成功）、琥珀 `#e0a458`（警告/过期）、红 `#d96a6a`（错误/上升）、蓝 `#6a9fd9`（信息）。
- **字体**：Inter（中文回退 PingFang SC / Microsoft YaHei）。
- **圆角**：8px（小 6px、大 12px）。
- **图标**：Lucide 风格线性 SVG（`stroke-width` 1.6），**不使用 emoji**。项目当前用 Naive UI 组件，图标库待落地时统一选型（建议 `@vicons` 或本地 SVG 组件）。

## 6. 范围与边界（YAGNI）

**本期包含**：

- 工作台框架 + 顶栏「工作台 / 智能问数」平级入口。
- 工作台首页（概览 + 场景卡片 + 任务列表 + 平台条）。其中"待处理预警""今日数据采集"两张统计卡先置灰待接入；场景卡仅"数据播报"可用，其余三张置为不可点击。
- 创建向导（模板向导 + AI 引导），第一版只需覆盖**数据播报**模板。
- 平台管理（凭据加密存储 + 多账号 + 操作提示词 + 平台技能 + 平台功能 + 自动重登）。
- 任务详情页 + 任务对话（复用 Hermes 对话运行时）。
- 智能问数顶栏入口与页面容器（接入数智大气现有对话服务，本项目不实现对话逻辑）。
- 场景模板系统（重构现有硬编码值守模板）。

**本期不包含 / 后续规划**：

- 集成数值大气的用户系统与权限系统（独立 spec）。
- 智能问数对话服务本身的实现（由数智大气提供，本项目只做入口与集成）。
- 数据采集 / 异常预警 / 报告生成场景（占位，模板后续补齐）。
- "待处理预警""今日数据采集"的数据采集、入库与持久化方案（尚未确定，先占位）。

## 附录：架构可行性结论（基于代码调研）

只改 Koa 服务器层 + Vue 前端层、不动 Hermes Agent，能否支撑以上功能？结论：**绝大部分可行**。判断法则——

- 功能需要 **LLM/工具去"想/做"事** → 必须走 Hermes（agent-bridge 或 Gateway）。
- 功能只是 **搬数据、调外部 API、管状态、做 UI 交互** → 纯 Koa + Vue 可控。

逐项：

| 模块 | 改哪层 | 依赖 Hermes |
|---|---|---|
| 工作台首页 | Vue + 少量 Koa 统计接口 | 否 |
| 场景模板系统 | Vue + 模板 CRUD API/store | 否 |
| 模板向导 | Vue 为主 | 否（终点仍是 `CreateJobRequest`） |
| AI 引导 | Vue + Koa，执行靠 Hermes | 是 |
| 平台管理（凭据/提示词/技能/功能存取） | Vue + Koa 加密存储 | 否 |
| 平台自动登录（用凭据真去登录） | — | 是（agent-browser），可行性待验证 |
| 智能问数 | Vue + 独立外部 HTTP 客户端 | 否（走外部服务） |
| 任务详情对话 | Vue + 复用 run-chat | 是（复用，不新造） |

两个硬约束：

1. **守护任务的调度与执行始终在 Hermes**：jobs 存于 Hermes profile 的 `cron/jobs.json`，由 Gateway(:8642) 触发，Koa 只是 `hermes cron` CLI 的包装器。向导无论多花哨，最终都是生成 prompt 写进 jobs，由 Hermes 执行。
2. **平台自动登录是唯一可行性存疑项**：存凭据纯 Koa 能做；"用凭据自动登录并维持会话"靠 agent-browser，需实际验证（验证码、会话持久化、Electron 打包后稳定性）。建议本期降级为"凭据集中存储 + 手动触发登录"，自动维持会话作为后续增强。

补充：Koa 层已有大量直连第三方 HTTP 的先例（STT/TTS、模型目录、OAuth），智能问数对接外部服务有现成模式，但需**新建独立外部 HTTP 客户端**，不能复用 `api/client.ts`（否则会把内部 token 泄露给外部端点、误触发硬编码的 `admin/123456` 刷新逻辑）。
